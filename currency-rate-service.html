Currency Rate Service

Autonomous gRPC Microservice for Exchange Rate Management

Personal Project by Igor Kudinov

Standalone Go microservice that autonomously collects, stores, and serves live and historical currency exchange rates to applications within the Expense Tracker ecosystem via gRPC. Addresses multi-currency visibility requirements and establishes the first inter-service communication pattern in the evolving microservice architecture.

Key Features

  • Autonomous polling engine with per-pair goroutines and configurable intervals
  • Primary + backup provider failover with automatic staleness flagging
  • gRPC API on :50052 — GetRate, GetRates (batch), ListSupportedPairs
  • Historical rate accumulation from day one (full insert history)
  • Database-driven config — pairs, providers, intervals managed via DB without restart
  • Generic JSON adapter — new REST providers added via config only (no code changes)
  • HTTP health endpoints on :8090 — liveness (/healthz) and readiness (/readyz)
  • Structured JSON logging (slog), gRPC reflection, graceful shutdown

Repository Structure

currency-rate-service — standalone microservice repository with independent deployment lifecycle.

/cmd/server — service entrypoint: config loading, DB connection, component wiring, graceful shutdown.

/internal/adapter — RateProvider interface, generic JSON adapter (URL templates + JSONPath), adapter registry.

/internal/polling — scheduler with per-pair goroutines, failover logic, pgx type converters.

/internal/grpc — gRPC server handlers + protoc-generated pb files (GetRate, GetRates, ListSupportedPairs).

/internal/health — HTTP health handler: /healthz (liveness) and /readyz (readiness + JSON status).

/internal/repository — sqlc-generated type-safe queries for all 5 tables.

/migrations — 5 golang-migrate migrations with seed data (pairs, providers, assignments, rates, health).

/proto — protobuf definition for CurrencyRateService.

/Documentation — BRD and SRS.

Architecture Overview

The service is composed of three internal subsystems:

  • Polling Subsystem — scheduler triggers rate collection per configured interval; adapter registry dispatches to generic JSON adapters (config-driven) or custom adapters (code); health tracker records per-provider status;
  • Data Layer — all business configuration (pairs, providers, intervals) lives in PostgreSQL; sqlc/pgx provide type-safe access; config loader reads pairs and providers at startup and on reload;
  • API Layer — gRPC server exposes GetRate, GetRates, and ListSupportedPairs; an HTTP health endpoint is available for infrastructure monitoring.

External providers (fawazahmed0, ExchangeRate-API, Frankfurter) are contacted only by the polling subsystem — gRPC consumers always read from the local database, guaranteeing sub-50ms response times.

Full architecture diagrams (component and sequence) are available in the SRS document.

Ecosystem Context

Currency Rate Service is part of the Expense Tracker ecosystem and serves as its first independent microservice:

  • Expense Tracker Backend (v0.3.0) — primary gRPC consumer; retrieves rates to display currency equivalents alongside transactions;
  • Flutter Mobile Client (v0.5.0) — indirect beneficiary via enriched transaction data from the backend;
  • Invoice Generator (future) — planned direct consumer for EUR↔RSD conversion in international invoices.

The service operates autonomously — it polls and stores rates regardless of consumer availability. If unreachable, consumers gracefully degrade by showing "rate unavailable" rather than stale data. This establishes the inter-service communication pattern documented in the Expense Tracker gRPC Integration TDD (ADR-6).
